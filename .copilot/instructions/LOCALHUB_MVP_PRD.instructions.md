# LocalHub MVP PRD

## 1. 문서 개요

### 1.1 목적
LocalHub는 공공데이터 기반 지역 정보 공유 커뮤니티 서비스다. 본 문서는 RFP에 명시된 필수 요구사항과 선택 기능 우선순위를 기준으로 MVP 범위, 사용자 흐름, 기능 요구사항, 기술 요구사항, 데이터 정책, 배포 기준, 수용 기준을 정의한다.

### 1.2 대상 독자
- 제품 기획자
- 프론트엔드 개발자
- 백엔드 개발자
- QA/테스트 담당자
- 발표 자료 작성 담당자

### 1.3 용어 정의
- 권역: 서울, 대전/충청, 구미/경북, 광주/전라, 부산 중 하나의 지역 묶음
- 제공 JSON: 발주사가 사전 수집·가공하여 제공하는 데이터 파일
- 익명 커뮤니티: 회원가입/로그인 없이 게시글을 작성·조회하는 방식
- 수정용 비밀번호: 게시글 작성 시 함께 저장되는 평문 비밀번호. 수정/삭제 시 일치 여부만 확인

---

## 2. 제품 개요

### 2.1 서비스 목표
- 제공 JSON만으로 지역 정보 탐색과 커뮤니티 참여가 가능한 MVP를 구축한다.
- 별도 인증 없이도 게시글 작성과 관리가 가능한 익명 게시판을 제공한다.
- 챗봇을 통해 관광지, 축제, 맛집, 게시글 검색을 자연어로 탐색할 수 있게 한다.
- 배포 완료 후 Netlify 프론트와 Render 백엔드 URL이 정상 동작해야 한다.

### 2.2 핵심 가치
- 빠른 초기 렌더링과 단순한 데이터 흐름
- 공공데이터 기반의 신뢰 가능한 지역 정보
- 익명 참여로 진입장벽 최소화
- 데모에서 한눈에 이해되는 홈/게시판/챗봇 구조

### 2.3 제품 범위 요약
- Must have: 데이터 연동, 커뮤니티 CRUD, 챗봇, Vue 3 SPA, FastAPI + SQLite, 환경변수 관리, 배포, 데이터 라이선스 문서화
- Should have: 커뮤니티 게시판 추가기능, 축제 캘린더, 데이터 시각화 대시보드, 지도 시각화
- Could have: 날씨 연동, 소셜 공유, 다국어 지원
- Won't have: 실시간 알림, 경로 안내

---

## 3. 문제 정의

사용자는 지역 정보를 찾고, 커뮤니티 글을 읽고, 질문을 올리며, 축제/관광/맛집 정보를 빠르게 탐색하고 싶어 한다. 하지만 외부 API를 직접 연결하거나 복잡한 인증 구조를 도입하면 개발·배포 난도가 높아지고, 발표용 데모 안정성이 떨어진다. 따라서 제공 JSON을 기반으로 단순하고 견고한 구조의 SPA + REST API 조합이 필요하다.

## 4. 목표 사용자

### 4.1 방문자
- 회원가입 없이 지역 정보를 탐색하고 게시글을 조회하는 사용자

### 4.2 작성자
- 게시글을 작성하고 이후 비밀번호로 수정/삭제하는 사용자

### 4.3 운영/발표 담당자
- 기능 명세, 데이터 출처, 배포 URL, WBS, 발표 자료를 제출해야 하는 담당자

## 5. 성공 기준

### 5.1 기능 성공 기준
- 선택한 1개 권역의 게시판에서 목록/상세/작성/수정/삭제가 동작한다.
- 수정/삭제는 작성 시 저장한 비밀번호 일치 시에만 가능하다.
- 챗봇은 `/api/chat`으로 제공 JSON 기반 질의에 응답한다.
- 홈 화면에서 최근 게시글, 큐레이션, 챗봇 진입이 자연스럽게 연결된다.
- 모바일 화면에서 챗봇이 플로팅 또는 전체 화면 방식으로 사용 가능하다.

### 5.2 제출 성공 기준
- 프론트 Git Repository URL 제출
- 백엔드 Git Repository URL 제출
- SQLite DB 파일 제출
- Netlify URL 제출
- Render URL 제출
- 기능 명세서에 데이터 출처·라이선스·수집일 포함
- WBS 및 발표 자료 제출

## 6. 기능 요구사항

## 6.1 Must Have

### 6.1.1 데이터 연동
#### 요구사항
- 제공 JSON 기반으로 프론트와 백엔드가 연동되어야 한다.
- 직접 공공 API 호출은 하지 않는다.
- 초기 데이터는 백엔드에서 SQLite에 적재하거나 서버 시작 시 로딩 가능한 형태로 관리한다.

#### 정책
- 데이터 원본은 발주사 제공 JSON으로 제한한다.
- 추가 수집 데이터가 필요할 경우 라이선스와 공공누리 유형을 사전 확인한다.
- 활용 데이터 목록에는 출처, 라이선스, 수집일을 포함한다.

#### 수용 기준
- 프론트는 API 응답만으로 화면을 렌더링한다.
- 백엔드는 JSON 파일과 SQLite 사이의 데이터 흐름을 책임진다.

### 6.1.2 커뮤니티 CRUD
#### 요구사항
- 익명 게시판으로 동작한다. 회원가입/로그인은 없다.
- 게시글 작성 시 제목, 내용, 수정용 비밀번호를 입력한다.
- 수정/삭제는 저장된 비밀번호 일치 여부로만 허용한다.
- 비밀번호는 평문 저장 및 비교로 구현한다. 이는 교육 목적의 의도된 설계다.
- 선정한 1개 권역의 게시판에 대해 목록/상세/작성/수정/삭제 API와 Vue 화면을 구현한다.

#### 게시글 기능 상세
- 목록 조회: 페이지네이션 또는 무한 스크롤 중 하나를 선택 가능
- 상세 조회: 게시글 본문, 메타 정보, 댓글 영역 표시 가능
- 작성: 제목, 내용, 비밀번호, 선택적으로 태그/이미지/카테고리 입력 가능
- 수정: 비밀번호 확인 후 제목/내용 변경 가능
- 삭제: 비밀번호 확인 후 삭제 가능

#### 추가 기능 우선순위
- 조회수
- 게시글 검색
- 좋아요
- 이미지 첨부
- 태그

#### 수용 기준
- 잘못된 비밀번호 입력 시 수정/삭제가 거부된다.
- 삭제된 게시글은 목록과 상세에서 노출되지 않는다.
- 목록 화면에서 게시글 작성으로 진입 가능하다.

### 6.1.3 챗봇
#### 요구사항
- FastAPI `POST /api/chat` 엔드포인트를 구현한다.
- 제공 JSON 기반 자연어 질의응답을 처리한다.
- 주요 질의 유형은 관광지 추천, 축제 일정, 맛집 위치, 게시글 검색이다.
- 채팅 UI는 대화 히스토리를 유지하고 모바일 대응이 가능해야 한다.
- 플로팅 버튼으로 진입할 수 있어야 한다.

#### 챗봇 동작 정책
- 질문 의도 분류 후 적절한 JSON 데이터 또는 DB 데이터에서 결과를 가져온다.
- 응답은 짧은 요약과 추천 목록 중심으로 제공한다.
- OpenAI API 사용은 가능하나, 필수는 아니다.
- 오프라인 데모 안정성을 위해 규칙 기반 또는 검색 기반 fallback을 둔다.

#### 수용 기준
- 사용자가 “추천해줘”, “일정 알려줘”, “맛집 찾아줘”, “게시글 검색해줘” 같은 요청을 입력했을 때 결과를 반환한다.
- 모바일에서 채팅창이 화면을 가리지 않고 사용할 수 있다.

### 6.1.4 프론트엔드
#### 요구사항
- Vue.js 3 기반 SPA 구조로 구현한다.
- 라우팅을 사용해 홈, 게시판 목록, 상세, 작성/수정, 챗봇 UI를 분리한다.
- 반응형 레이아웃을 제공한다.

#### 수용 기준
- 새로고침 후에도 라우트가 유지된다.
- 모바일 폭에서 레이아웃이 깨지지 않는다.

### 6.1.5 백엔드
#### 요구사항
- FastAPI 기반 REST API 서버를 구축한다.
- SQLAlchemy ORM을 적용한다.
- SQLite 스키마를 설계한다.
- 예시 엔터티는 posts, comments, locations 등을 포함한다.
- 민감 정보는 `.env`로 관리하고 소스코드에 직접 기재하지 않는다.
- `.env`는 `.gitignore`에 포함한다.

#### 수용 기준
- API 서버가 SQLite 파일을 사용해 게시글 데이터를 CRUD로 관리한다.
- 환경변수 미설정 시 명확한 오류를 반환한다.

### 6.1.6 배포
#### 요구사항
- 프론트엔드는 Netlify에 배포한다.
- 백엔드는 Render에 배포한다.
- 배포 완료 후 각 서비스 URL이 정상 동작해야 한다.

#### 수용 기준
- 배포 URL이 제출 자료에 포함된다.
- 프론트에서 백엔드 API 호출이 배포 환경에서도 정상 동작한다.

### 6.1.7 데이터 라이선스 문서화
#### 요구사항
- 추가 수집 데이터가 있으면 공공누리 유형을 확인한다.
- 기능 명세서에 활용 데이터 목록을 별도 섹션으로 기재한다.
- 각 데이터 항목에 출처, 라이선스, 수집일을 포함한다.

#### 수용 기준
- 문서만 보고도 데이터 사용 가능 여부를 추적할 수 있다.

## 6.2 Should Have

### 6.2.1 커뮤니티 추가 기능
권장 구현 순서:
1. 검색
2. 조회수
3. 좋아요
4. 태그
5. 이미지 첨부

#### PRD 반영 이유
- 기존 CRUD를 크게 흔들지 않고 가시적 임팩트를 높일 수 있다.
- 발표 데모에서 사용자의 참여감을 높인다.

### 6.2.2 축제 캘린더
- FullCalendar를 사용해 권역별 축제 일정을 시각화한다.
- 제공 JSON의 축제 데이터를 재가공하여 달력에 표시한다.
- 날짜 클릭 시 축제 상세를 보여줄 수 있다.

### 6.2.3 데이터 시각화 대시보드
- Chart.js 또는 D3.js 기반으로 지역별 게시글 수, 인기 키워드, 방문자 추이 같은 집계를 시각화한다.
- 홈 화면 혹은 별도 대시보드 탭에 배치한다.

### 6.2.4 지도 시각화
- Leaflet 또는 Kakao Maps로 관광지·맛집 핀을 표시한다.
- 권역 필터를 제공한다.
- 좌표가 없는 데이터는 우선순위에서 제외하거나 추후 보완한다.

---

## 6.3 Could Have

### 6.3.1 날씨 정보 연동
- 외부 날씨 API 키가 필요할 수 있으므로 발주사 제약을 확인한 뒤 착수한다.

### 6.3.2 소셜 공유
- 카카오톡 공유, 링크 복사, OG 태그를 고려한다.
- 단, 앱키와 도메인 등록 작업이 필요하다.

### 6.3.3 다국어 지원
- 한/영 전환을 제공할 수 있으나 번역 범위가 커서 우선순위는 낮다.

## 6.4 Won't Have

### 6.4.1 실시간 알림
- WebSocket 기반 알림은 이번 스프린트 범위에서 제외한다.

### 6.4.2 경로 안내
- 복수 지점을 연결하는 경로 최적화는 난이도와 리스크가 높아 제외한다.

## 7. 사용자 시나리오

### 7.1 홈 방문자
1. 사용자가 홈에 접속한다.
2. 선정 권역 소개 배너, 최근 게시글, 큐레이션, 챗봇 진입 버튼을 확인한다.
3. 관심 있는 게시글 또는 챗봇으로 이동한다.

### 7.2 게시글 작성자
1. 게시판 목록에서 글쓰기 버튼을 누른다.
2. 제목, 내용, 비밀번호를 입력해 게시글을 등록한다.
3. 상세 화면에서 게시글을 확인하고 수정 또는 삭제를 시도한다.
4. 비밀번호가 일치하면 변경 사항이 반영된다.

### 7.3 챗봇 사용자
1. 플로팅 버튼을 눌러 챗봇을 연다.
2. “서울에서 주말에 갈 만한 축제 알려줘”처럼 자연어로 질문한다.
3. 챗봇이 제공 JSON과 DB 데이터를 바탕으로 응답한다.

## 9. 시스템 아키텍처

### 9.1 프론트엔드
- Vue 3 SPA
- Vite 기반 개발 서버
- 상태 관리는 단순화하고 API 호출 중심으로 구성

- FastAPI REST API
- SQLAlchemy ORM
- SQLite DB
- CORS 설정 필요

### 9.3 데이터 흐름
1. 제공 JSON을 백엔드가 읽는다.

3. 프론트는 API로 데이터만 요청한다.
4. 챗봇은 DB 또는 JSON 검색 결과를 자연어로 응답한다.

### 9.4 권장 테이블
- posts
- comments
- locations
- festivals
- categories
- chat_history
- metadata 또는 data_sources

## 10. 비기능 요구사항

### 10.1 성능
- 홈 화면은 초기 렌더링이 빠르게 느껴져야 한다.
- 목록 조회 API는 페이지네이션 또는 제한된 개수로 응답한다.

### 10.2 보안
- API 키, DB 경로, 외부 인증 정보는 `.env`로 분리한다.
- 저장소에 비밀값이 노출되지 않도록 한다.
- 게시글 비밀번호는 교육 목적상 평문 저장 정책을 따른다.

### 10.3 유지보수성
- 프론트와 백엔드 책임을 분리한다.
- 데이터 출처와 라이선스는 별도 문서화한다.
- 하드코딩된 상수는 최소화한다.

### 10.4 배포 안정성
- Netlify와 Render 배포 후 동일한 API 계약을 유지한다.
- 개발/배포 환경의 엔드포인트는 환경변수로 분리한다.

## 15. 위험 요소 및 대응

### 15.1 위험 요소
- 제공 JSON 스키마가 복잡하거나 화면 요구와 맞지 않을 수 있다.
- 챗봇 응답 품질이 데이터 검색 방식에 따라 흔들릴 수 있다.
- 지도 기능은 좌표 데이터 정제 비용이 높다.
- 배포 환경에서 CORS 또는 환경변수 설정 오류가 발생할 수 있다.

### 15.2 대응 방안
- 초기에 JSON 샘플을 검토하고 공통 스키마를 확정한다.
- 챗봇은 rule-based fallback을 둔다.
- 지도는 좌표가 준비된 데이터만 우선 노출한다.
- 배포 전 로컬/스테이징에서 환경변수를 점검한다.
